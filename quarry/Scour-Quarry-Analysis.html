Scour.ing × Quarry

What scour.ing built, how they did it, what Quarry should steal, and where Quarry is already ahead.
Scour: 1 engineer · self-funded · 2,919 users · 24,546 sources Quarry: personal-first · RSSHub-backed · knowledge garden
Core philosophy

Scour.ing

Hidden gems over viral content
Surfaces content you'd never find otherwise — indie blog with 50 readers can beat a viral post
Deliberately uses zero popularity signals — no vote counts, shares, or backlinks factor in
Ranking = semantic embedding distance from your stated interests (binary quantized embeddings + Hamming distance)
Explicit interests only — one click on a cooking article won't flood you with cooking content
24,546 community-sourced feeds · 1.68M domain blocklist · filters listicles + emoji-led titles
Source diversity enforced — no single source dominates

Quarry

1hr/day intelligence compression
Surfaces what's hitting today in your quest areas — you want to know what's moving, not just what's obscure
Vote-weighted scoring: platform votes × source reliability + saves — popularity is a useful signal for bug bounty/AI/ARR content
Quest-based intent buckets — not just interests, but named missions with tracked progress
Capture layer: reading → annotation → memo garden → spaced recall. Scour stops at reading.
RSSHub as feed backbone — 1,000+ routes, self-hostable, no community dependency
Discipline metric: streak tracking. Reading is a practice, not just discovery.
Ranking engine — side by side

Scour.ing — semantic embeddings

score = cosine_sim(article_embedding, best_interest)
+ recency_bucket_adjustment
+ source_prominence_adjustment (from reactions)

// No platform votes used at all
// Binary quantized → Hamming distance
// Real-time, not pre-computed
Reactions (Love/Like/Dislike) make gradual, confidence-weighted adjustments to source prominence and topic balance. They don't add new topics. Decay over time prevents any source from dominating permanently.

Quarry — vote-weighted formula

score = (platform_votes × src_reliability
       + saves × 3) / (age_h + 2)^0.7

where src_reliability = f(thumbs_history)

// Transparent, visible, hoverable
// No embedding model or API needed
Thumbs △/▽ adjust source reliability over time. Saves carry 3× weight (action > click). Age decay at 0.7 exponent keeps older-but-relevant content in view longer than HN's aggressive gravity. Score visible on hover.

Interest / topic definition
 ✓ free-form text Quests (named buckets) Scour: type anything → embeddings. Quarry: named missions with color + count. Quarry's model is more intentional; Scour's is lower friction.
Reaction granularity

Keyboard navigation
 j/k/o/s/a/l/d/x + go-to shortcuts Not implemented j/k to move, o to open, s to save, a=love, l=like, d=dislike, x=not interested. This is table stakes for a reading tool. Fast to add.
Timeframe filter

Deduplication

Source diversity enforcement

"Why didn't I see this?"

OPML import
 Import + infer interests Not implemented Bring your existing RSS subscriptions. Medium effort. High value for onboarding (the "setup in 2 min" hook).
Email digest
 Daily / weekly Not implemented Aligns perfectly with "Daily Radar" concept. The email IS the radar for days you don't open the app. Habit adhesion mechanism.
Reading history
 Private · "seen" filtering Not implemented Auto-hide seen items by default. "Show seen" toggle. History tab. Find posts you forgot to bookmark. Very fast to add.
Knowledge capture
 Stops at Likes page Memo garden · annotation · tags Quarry's biggest differentiator. Scour has no writing layer. Quarry: read → annotate → memo → recall. This is the entire "Memos becomes junk drawer" defense.
Spaced recall
 Not implemented Recall tab · weekly digest Quarry wins. Scour has no memory layer. "Revisit in 7 days" is the anti-junk-drawer mechanism Scour entirely lacks.
Side-by-side reader + writer
 No writing surface Read left · write memo right Quarry wins. Forces note-taking as part of reading flow. The "O'Reilly daily radar" pattern.
Discipline tracking
 Not implemented 21-day streak grid · 1hr drill Quarry wins. Scour has no consistency mechanic. The streak answers Khabib's question.
Keyboard shortcuts
 Comprehensive None This is the fastest win. j/k navigation alone makes Quarry feel like a real reader.
Source management

Mobile / PWA

Not interested signal

Replace △/▽ with ♥ Love (saves to garden + strong signal), ↑ Like (soft signal), ↓ Dislike (drops source reliability). Love ≠ Save — Love is "this shaped my thinking"; Like is "good, move on." More signal gradient for the scoring formula.

Auto-track items you've clicked or reacted to. Hide them from the feed by default. Toggle "show seen" in filter pills. Keeps the radar fresh every session — you never re-read what you've already processed. Reading history tab auto-generates from this.

Priority 1 · High value · Low effort

Fresh (default, last 24h with recency spread) / 3d / 1w / 1m. Swap the current "new/saved/high" pills for time-windowed views. Fresh mode: show a mix of recent items, not just newest-first. Prevents the "only the last hour matters" trap.

Priority 1 · Medium value · Low effort
Content quality filters
Toggle to filter: listicle titles ("5 Ways to…", "Top 10…"), emoji-led titles, duplicate domain items. Simple regex + a settings panel. Scour runs this before ranking on 1.68M domains — Quarry runs it per-session client-side. Same effect, zero server.
Effort~20 min (regex filter in items memo)
Priority 1 · Medium value · Low effort
Source diversity cap
Max 2 items from any single source per feed page. Prevents one highly-active source (e.g. HN) from crowding out everything else. Already have source tracking — just add a per-source counter during filter/sort.

"Why didn't I see this?" tool
Paste any URL → Quarry checks: Is the domain in any active source? Is the score below your floor? Was it deduped away? Was it filtered by quality rules? Great for trust, great for debugging your source setup. Shows the scoring formula is working.

What NOT to adopt from scour.ing
FeatureWhy scour.ing has itWhy Quarry should skip
Zero popularity signals
 Their philosophy is hidden gems over viral. Makes sense for breadth-of-web discovery. Your quests (bug bounty, ARR, AI disruption) benefit from knowing what's getting traction NOW. A $50k bounty report with 600 upvotes IS the signal. Don't suppress it.
Community-sourced feeds (24,546)
 Discovery across the whole web. Social proof. Community cold-start is solved by crowd. Personal-first. RSSHub gives 1,000+ routes you control. Community sources add noise and trust issues. You're curating your signal, not the crowd's.
Embedding-based semantic matching
 Finds relevant content even when phrased differently. Works for niche, obscure interests. Your interests are concrete quest names with 10 example items each (per the strategic plan). Keyword + vote scoring works. If you ever want embeddings, use a local ONNX model client-side — no API.
Public Likes page + RSS feed of Likes
 Community discovery. Others subscribe to your taste. Personal tool, not a publishing business yet. Greene's advice (your filtered output as product) is the right long-term direction — but that's V2, not today.
Daily email digest · Quarry architectureyour design, not scour.ing's
Scour.ing digest
Weekly, Fridays at 10 UTC
Top 10 posts from past week · relevance-ranked · reaction buttons in email · interest recommendations at bottom. One digest. One cadence. No categories.
Quarry digest · your concept
Daily, 7am · 5–7 newsletters by category
Routine fetches all sources → Claude summarizes per category → sends category-specific newsletters. Each is a mini-radar for that quest thread.

Routine architecture
Step 1
Fetch all RSSHub routes → parse items → deduplicate by URL normalization
Step 2
Classify each item into a category via title/tags/source heuristics (no API needed)
Step 3
Claude reads top N per category → writes a 3–5 sentence synthesis (what matters, why now)
Step 4
Send 5–7 emails (one per non-empty category) · reactions link back to Quarry app
Trigger: cron at 06:45 UTC daily · MCP or REST fetch · Claude haiku (cheapest) for summarization · only AI cost in the entire stack

Verdict

Scour.ing strengths

✓
Semantic matching finds things you didn't know to search for
✓
Community sources = zero setup, immediate breadth
✓
Keyboard shortcuts + reading UX feel native
✓
No popularity bias — genuinely surfaces obscure quality
✓
Quality filtering (1.68M blocklist) is a major moat

Quarry advantages

●
Memos + recall: turns reading into knowledge that compounds
●
Quests = named intent, not vague "interests"
●
Streak + discipline layer — Scour has no consistency mechanic
●
Transparent scoring — you know exactly why something ranked
●
RSSHub: full control, no community cold-start, self-hostable
